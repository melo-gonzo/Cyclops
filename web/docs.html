What it is

%DEVICE_NAME% is a trigger-based audio + video capture node built on the Seeed XIAO ESP32-S3 Sense (the firmware also builds for the ESP32-CAM, video only). It listens and watches; when sound or motion crosses a threshold it saves a short clip — with the seconds before the event included — to the SD card.

Everything is served from an on-device web dashboard over your LAN or the device's own WiFi hotspot. There is no cloud: capture, storage, and playback are entirely local.

Capture pipeline

Audio: PDM mic → I2S DMA → DSP (high/low-pass filters, RMS level, FFT) → trigger decision → PSRAM pre-roll ring → SD writer → clip_NNNNN_x.wav.

Video: camera → JPEG frame ring (PSRAM) → motion frame-diff → AVI writer → vclip_NNNNN_x.avi. An audio trigger can also record a paired video clip and vice-versa.

Concurrency & storage

  • Independent FreeRTOS tasks (audio capture, camera, MJPEG stream, SD writer, web server) pinned across both cores.
  • All SD/SPI access is serialized through one mutex. The clip list and cap-eviction hot paths read an in-RAM index of /clips instead of scanning the card, so a slow card never stalls the UI.
  • Settings live in NVS and survive reboots. The level / heatmap / metrics history is mirrored to SD, so the plots survive power loss too.

Clip storage & retention

  • Clip numbers run 00000–09999 and wrap, keeping names tidy. Each kind (audio, video) keeps the newest N; the oldest is evicted first (wrap-aware).
  • Per-clip over-threshold data (level / threshold / Hz) is persisted in clipmeta.csv so the Clips table's "Over thr" column survives reboots.
  • Continuous segments (Record page) roll off by a size/time retention cap, independent of triggers.

Code shape

"Functional core / imperative shell": the decision math — filters, thresholds, FFT, retention, parsing — lives in pure C++ headers unit-tested on the host (pio test -e native). The .cpp shells own the hardware and timing. XIAO-only subsystems are compiled out of the ESP32-CAM build, so this page always matches the running firmware.

Level trigger — adaptive

The device continuously learns the room's noise floor with a slow-rise / fast-fall estimator: it follows a sudden quieting quickly but creeping ambient drift slowly. The trigger fires when the level exceeds floor × sensitivity, clamped up to the "min threshold" so a silent room can't become a hair-trigger.

A sustained loud sound is gradually absorbed into the floor (habituation) so it stops re-firing, while brief transients keep triggering. A manual threshold (> 0) overrides the adaptive value entirely.

Level trigger — statistical

An alternative model that tracks the running mean and standard deviation (σ) of the level over a window (in minutes) and fires on samples k·σ above the mean. A busy, variable room raises its own bar automatically; a quiet room stays sensitive. One knob: k (lower = more sensitive). k is the σ multiplier — distinct from the measured σ shown live.

Spectral (band) trigger

Runs an FFT and learns a per-band floor with the same adaptive model. It fires when any single frequency band crosses its own threshold — catching narrow-band sounds (beeps, alarms, whines) that broadband level misses — and names the hottest band in the log and tooltip.

Motion & pre-roll

Motion: an on-device block frame-diff; a video clip records when enough of the frame changes. Off by default because it costs CPU.

Pre-roll: a rolling buffer continuously holds the last few seconds, so every clip starts before the trigger. Gain: thresholds scale with mic gain, so changing gain doesn't change how sensitive the trigger feels.

First boot

Out of the box the device raises a WiFi hotspot. Connect to it, open the dashboard, and on the WiFi page enter your home network. Set a hostname — the device is then reachable at <host>.local via mDNS — and optionally a static-IP last octet, which lets a whole fleet run one identical binary while staying individually addressable. Reboot to apply network changes; other settings apply live.

Accounts

Admin
Full control: every setting, clip deletion, firmware updates.
Viewer (optional)
Read-only: can watch, listen, and download clips, but cannot change any setting. Enable it on the WiFi page.

Firmware (OTA)

Update over the air from the Firmware card on the WiFi page: upload a firmware.bin and the device verifies it, writes the spare OTA partition, and reboots into it. The version line shows the running build. The dual-OTA layout means a bad image can roll back to the previous one.
